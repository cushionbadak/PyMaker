<p>I found the Python 3 documentation on <a href="https://docs.python.org/3/library/functions.html#chr" rel="nofollow noreferrer"><code>chr</code></a> and <a href="https://docs.python.org/3/library/functions.html#ord" rel="nofollow noreferrer"><code>ord</code></a> to be a little unclear as to how they relate to the two main textual data types: <code>str</code> and <code>bytes</code>. Or maybe I'm overthinking it.</p>
<p>Here is what I think probably happens, but can you let me know if I'm right?</p>
<ul>
<li><code>ord()</code> takes as input a single-character <code>str</code> and returns an <code>int</code>. The input is a <code>str</code> just like any other <code>str</code> in Python 3. In particular, it is <strong>not</strong> <code>bytes</code> encoded in some specific Unicode format like UTF-8, rather it represents Unicode Code Points in Python's internal <code>str</code> format.</li>
<li><code>chr()</code> takes as input an <code>int</code>, and returns a single character <code>str</code>. The returned <code>str</code> is just like any other <code>str</code> in Python, and similarly is not a specific encoding using <code>bytes</code>.</li>
<li>At no point do either <code>ord()</code> or <code>chr()</code> deal with <code>bytes</code>, nor do they deal with specific Unicode formats like UTF-8, they are only dealing with Python's internal <code>str</code> representation which deals more abstractly with Unicode Code Points.</li>
</ul>

<div class="post-text" itemprop="text">
<p>You are right.</p>
<p><code>ord()</code> and <code>chr()</code> deal only with single-character strings.</p>
<p>Their documentation is quite clear about that:</p>
<pre><code>&gt;&gt;&gt; help(ord)
ord(c, /)
    Return the Unicode code point for a one-character string.

&gt;&gt;&gt; help(chr)
chr(i, /)
    Return a Unicode string of one character with ordinal i; 0 &lt;= i &lt;= 0x10ffff.
</code></pre>
<p>Use <code>str.encode</code> / <code>bytes.decode</code> for conversion to/from bytes.</p>
