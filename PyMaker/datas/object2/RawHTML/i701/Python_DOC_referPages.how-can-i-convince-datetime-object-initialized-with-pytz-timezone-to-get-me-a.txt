<p>In one part of program timezone aware time object is created. In other part it is necessary to get timezone identifier back.</p>
<p>To avoid XY issues: I am calculating sunrise and sunset data using <code>skyfield</code> library. To do this I need to pass timezone as one of parameters.</p>
<p>From what I see at <a href="https://docs.python.org/3/library/datetime.html" rel="nofollow noreferrer">https://docs.python.org/3/library/datetime.html</a> there is a <code>timezone</code> but for setting timezone, not getting it.</p>
</div>
<div class="post-text" itemprop="text">
<p>The TZ database timezone name is stored as the <code>zone</code> attribute of the zone object returned by <code>pytz.timezone()</code>:</p>
<pre><code>&gt;&gt;&gt; import pytz
&gt;&gt;&gt; import datetime
&gt;&gt;&gt; 
&gt;&gt;&gt; timezone = pytz.timezone('Poland')
&gt;&gt;&gt; date = timezone.localize(datetime.datetime(2018, 10, 1))
&gt;&gt;&gt; date.tzinfo.zone
'Poland'
</code></pre>
<p>As you can see, the zone object itself is available as <code>date.tzinfo</code> after calling <code>localize()</code>, so you can just use that directly instead of passing the name back into <code>pytz.timezone()</code> if that's why you need it.</p>
</div>
<span class="comment-copy"><code>Poland</code> is shorter and more readable than <code>Europe/Warsaw</code> so I will keep using it as long as it works.</span>
<span class="comment-copy">According to the <a href="https://en.wikipedia.org/wiki/List_of_tz_database_time_zones" rel="nofollow noreferrer">list of timezone names in Wikipedia</a> <code>Poland</code> is deprecated, replaced by <code>Europe/Warsaw</code></span>
<span class="comment-copy">The compiler doesn't care about the length and using the standard name ensures you won't have to check whether it still works, nor have to fix code you've forgotten about once it breaks. It also means you won't have to map timezone names that come from other libraries, files or programs, perhaps even your own</span>
