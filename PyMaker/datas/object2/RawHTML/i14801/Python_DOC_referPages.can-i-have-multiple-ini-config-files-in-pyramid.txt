<p>I'd like an equivalent of the <a href="https://speakerdeck.com/jacobian/the-best-and-worst-of-django?slide=81" rel="noreferrer">Django One True Way</a> settings layout: a shared base file, and then a production file and a development file, each of which import the shared base.</p>
<p>Is this possible with Pyramid's config?</p>
</div>
<div class="post-text" itemprop="text">
<p>Yes that's possible. In one of my projects I have a <code>production_base.ini</code> file and all other production inis inherit from it: </p>
<p><code>production_base.ini</code></p>
<pre><code>[app:main]
use = egg:xxx
maintenance_mode = False
</code></pre>
<p><code>production_www.ini</code></p>
<pre><code>[app:main]
use = config:production_base.ini
maintenance_mode = True  # overwrites the value in the base ini
</code></pre>
<p>You can also check <a href="http://pastedeploy.readthedocs.io/en/latest/index.html?highlight=config%3A#applications" rel="nofollow noreferrer">paste docs</a> for more examples.  </p>
<p>Side note - you can't inherit logging section though.</p>

<span class="comment-copy">I can confirm @matino's side note: <code>use =</code> will not work in the sections related to logging. The <a href="https://docs.python.org/3/library/configparser.html" rel="nofollow noreferrer">configparser</a> module will raise a <code>KeyError</code> for keys you'd expect to be imported (e.g. <code>keys</code>, <code>class</code>). Looks like the inheritance is a PasteDeploy feature, and not generally supported in <code>.ini</code> files.</span>
