<p>virtualenvwrapper-powershell interacts nicely with Python 2.7.
(<a href="https://pypi.python.org/pypi/virtualenvwrapper-powershell/2.7.1" rel="nofollow noreferrer">https://pypi.python.org/pypi/virtualenvwrapper-powershell/2.7.1</a>)</p>
<p>So, I suggest to install that as the system python AND then install Python 3.5.2 outside of the path.</p>
<p>Following steps:</p>
<ol>
<li><p>Install Python 2.7 <strong>ON</strong> the path</p></li>
<li><p>Install Python 3.5.2 <strong>OFF</strong> the path</p></li>
<li><blockquote>
<p>pip</p>
</blockquote></li>
</ol>
<blockquote>
<p>pip install --upgrade setuptools</p>
<p>pip install ez_setup</p>
</blockquote>
<ol start="5">
<li><blockquote>
<p>pip install virtualenv</p>
</blockquote></li>
<li><blockquote>
<p>pip install virtualenvwrapper-powershell</p>
</blockquote></li>
</ol>

<span class="comment-copy">FYI, that Ctrl+C exits Python's REPL is due to a bug in Windows 8 and 10, in which <code>ReadFile</code> doesn't set the last error to <code>ERROR_OPERATION_ABORTED</code> the way it's supposed to. Python's shell is supposed to default to raising a <code>KeyboardInterrupt</code> exception for Ctrl+C. To exit normally, as intended, type Ctrl+Z and press enter, or to kill Python type Ctrl+Break. This is what will continue to work when you upgrade to Python 3.6, which switched to calling <code>ReadConsole</code>, which works correctly.</span>
<span class="comment-copy">The Windows installer gives you the option to install pip, which you should then upgrade to the latest version via <code>python -m pip install --upgrade pip</code>.</span>
<span class="comment-copy">Python 3 has the <a href="https://docs.python.org/3/library/venv.html" rel="nofollow noreferrer"><code>venv</code></a> module. Give that a try.</span>
<span class="comment-copy">As I say in <a href="http://djangodeployment.com/2016/11/01/virtualenv-demystified/" rel="nofollow noreferrer">djangodeployment.com/2016/11/01/virtualenv-demystified</a>, I have some doubts as to how well virtualenvwrapper is supported on Windows. It may be better to use plain virtualenv.</span>
